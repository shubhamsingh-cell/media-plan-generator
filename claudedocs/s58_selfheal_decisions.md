## Trade-off notes on each

**1. Self-upgrade**: `app.py:5648` was already corrected in S56. The remaining false claims live in `docs/v3.1_system_state.md` (line 368 "weekly self-upgrade cycle", line 485 "weekly self-upgrade") and `docs/generate_docx.py` (lines 415, 561). These docs regenerate the `Joveo_Media_Plan_Generator_Product_Overview.docx` and `Nova_Chatbot_Architecture.docx` artifacts shared with Cindy and Anirudh. **Option A** would mean building an actual weekly loop (~600-800 LOC: job scheduler + test diff + prompt-golden regeneration via Haiku + Supabase metric emit) -- high cost, low user ROI since manual `/nova-test` + `/deploy-check` already cover 95%. **Option B** wins on honesty and shipping cost.

**2. resilience_router**: 2,354 LOC of well-written but fully dormant code. Callers required for Option A migration: `api_enrichment.py` has 13 retry-style try/except blocks (grep: "for.*in.*range.*retries|retry_count"), `nova.py` has ~8 external API call sites. Migrating would mean rewriting ~20 call sites to go through `resilient_fetch("web_scraping", "scrape", url=...)` etc. That's ~60-80 LOC of call-site changes + rollback risk (every wrapped call adds lookup + fallback-chain overhead). **Option B** deletion is the YAGNI/KISS answer. The functionality it promises (retry + circuit breaker + tier fallback) is 80% delivered by existing `circuit_breaker_mesh.py` (grep: it exists as a separate module) + individual `try/except` blocks in callers. Keeping it means every future engineer wonders "should I use this?" and the answer is always "no."

**3. anomaly_detector**: The module is clean, thread-safe, and has graceful degradation (startup grace, sample floors, alert cooldown). The `record_metric(METRIC_REQUEST_LATENCY)` wiring works. Adding the other 3 metrics is ~12 LOC per S56 line 88. Wiring to `alert_manager.send_alert()` on `is_anomaly=True` is ~10 LOC in the existing `monitoring.py` alert bridge (where `start_alert_bridge` already runs every 60s). **Option A** makes the module pull its weight. **Option B** (simplify to latency-only) discards good-quality work and a real product need (mem + error rate visibility matters). Keep it, wire it.

**4. Alert pipe**: This is THE unlock. Once `SLACK_ALERTS_WEBHOOK_URL` is set, FIVE existing callers immediately start delivering to Slack instead of `/tmp/nova_alerts.log`. Current state: Resend Tier 1 works (Shubham gets emails), but if Resend has an outage or if `RESEND_API_KEY` quota is exhausted, Tier 2 silently falls to a log file that's wiped on Render restart -- meaning real alerts during a real incident would vanish. **Option A** is 0 LOC, 5-minute fix (create #nova-alerts channel, add incoming webhook, set env var via Render API). **Option B** (Supabase table) is worse because the common failure case is "Supabase is slow/down" -- if the thing we're alerting about IS Supabase, the alert can't be written. Fallback chains must not depend on the components they're protecting.

## Strongest Recommendation

**Do item 4 first** (set `SLACK_ALERTS_WEBHOOK_URL`, 0 LOC, 5 min). It unlocks real alerting for 5 callers immediately, has zero risk, and turns items 1-3 from "abstract cleanup" into "decisions backed by live signal." After #4 is live for 48h, do **#3** (wire 3 missing anomaly metrics + alert bridge, ~25 LOC) to get real coverage. Then batch **#1 + #2 as a single deletion PR** (~2,485 LOC removed, all doc truth-up + dead code).
